Plugin identity:
- Composer name: `logingrupa/oc-goodsreceived-plugin`
- October plugin code: `Logingrupa.GoodsReceivedShopaholic`
- License: MIT
- Tech: October CMS v4 + Lovata Toolbox ^2.2 + Lovata Shopaholic ^1.32 + PHP ^8.3

4 Settings toggles (Backend → Settings → Goods Received):
- `enabled` (switch, default off) — Master toggle
- `auto_deactivate_on_zero` (switch, default off) — When offer.quantity hits 0, set offer.active=false (skipped on offers where active_managed_by='operator')
- `auto_activate_on_stock` (switch, default off) — When inbound stock arrives for inactive offer, set offer.active=true (same operator-skip rule)
- `allow_initial_reset` (switch, default off) — Show the one-shot baseline-reset checkbox on upload preview

4 split permissions (Backend → Settings → Administrators → Roles → tab "Goods Received"):
- `logingrupa.goodsreceived.upload_invoices` — required to upload .HTM files
- `logingrupa.goodsreceived.apply_invoices` — required to click Apply
- `logingrupa.goodsreceived.override_invoices` — required to use override-and-reimport on a duplicate
- `logingrupa.goodsreceived.run_initial_reset` — required to trigger the one-shot reset

Console command (UI-11):
- `php artisan goodsreceived:recompute_active_from_stock {--chunk=500}`
- Calls `ActiveFlagService::reconcileAll()` chunked
- Honors `active_managed_by='operator'` skip
- Returns 0 on success, 1 on Throwable

5 Log::* event keys (operator troubleshooting):
- `goodsreceived.apply` (StockApplyService writes — units_added, offers_touched, applied_by, correlation_id)
- `goodsreceived.parse` (HtmInvoiceParser outcomes — invoice_number, total_lines, matched_lines)
- `goodsreceived.reject` (DuplicateInvoiceException + InvalidInputException + MalformedHtmException — invoice_number_attempted, prior_apply_at, reject_reason)
- `goodsreceived.initial_reset` (InitialResetService — invoice_id, offers_zeroed_count, products_deactivated_count)
- `plugin.boot.config_warning` (Plugin::boot self-check — max_file_uploads / upload_max_filesize warnings)

Override-and-reimport (D12, locked semantics):
- Add-on-top: re-apply increments offer.quantity additively on top of prior apply (NOT decrement-then-reapply, NOT delta calculation)
- Operator types literal "OVERRIDE" (uppercase, server-side ===) to confirm
- New Invoice row created with `override_of_invoice_id` pointing at prior + invoice_number suffixed `-OVR-<priorId>`
- Warning copy verbatim: "This re-applies the invoice ADDITIVELY on top of the prior apply. Stock will be incremented by new line quantities. This is NOT a delta calculation."

Initial-reset (UI-08, locked semantics):
- Visible only when `allow_initial_reset=true` AND no prior `Invoice.initial_reset_applied=true` row exists (one-shot)
- Operator types literal "RESET" (uppercase, server-side ===) to confirm
- Pre-mutation snapshot count surfaced in modal: "This will zero out N offers and deactivate M products"
- All offer.quantity → 0, all offer.active → false, all product.active → false
- Snapshot table `logingrupa_goods_received_initial_reset_snapshot` captures `prior_quantity`, `prior_offer_active`, `prior_product_active` for every offer (rollback feasibility — manual rollback runbook section)

GRN-canonical stock writer dependency (D-13, locked):
- This plugin CLAIMS ownership of `lovata_shopaholic_offers.quantity`
- The operator MUST manually disable quantity import in `Logingrupa.ExtendShopaholic` 1C XML config
- This plugin does NOT auto-disable that setting — D-13 explicit out-of-band

Multi-site (D-04):
- Same plugin code on .no / .lv / .lt servers
- Per-server DB → Settings naturally per-server (October Settings model)
- A toggle on .no does not affect .lv (verified manually per OPS-06)

    7. **## Initial-reset runbook (one-shot baseline)**
       - Pre-flight checklist:
         1. Confirm `Settings.allow_initial_reset` is ON for THIS site
         2. Confirm no prior reset has been recorded (visible in Invoice History — no row with `initial_reset_applied=true`)
         3. Estimate snapshot count: how many offers + products will be affected? The pre-mutation modal will show actual counts before commit.

       - Procedure:
         1. Upload first `.HTM` invoice
         2. Tick "One-shot baseline reset" checkbox on the preview screen
         3. Modal: pre-mutation snapshot count appears ("This will zero out N offers and deactivate M products")
         4. Type literal `RESET` (uppercase) to confirm
         5. Click "Run reset + apply" — the plugin runs in this strict order: PARSE → RESET → APPLY (per D-24)
       - What gets zeroed/deactivated:
         - All `lovata_shopaholic_offers.quantity` → 0
         - All `lovata_shopaholic_offers.active` → false
         - All `lovata_shopaholic_products.active` → false
         - The Invoice row gets `initial_reset_applied=true` so the gate cannot fire again
       - Snapshot table reading (rollback):

    8. **## GRN-canonical stock writer — disabling 1C XML quantity import (D13)**
       - Header: "**This step is required before turning the plugin on.**"
       - Explanation: Two import paths could write to `lovata_shopaholic_offers.quantity` — this plugin (the new GRN canonical writer) AND the existing `Logingrupa.ExtendShopaholic` 1C XML import. Per D-13, this plugin OWNS the `quantity` column. The operator MUST manually disable quantity-write in ExtendShopaholic's 1C XML config so the two writers don't fight.
       - Steps:
         1. Backend → Settings → Extend Shopaholic → 1C XML Import (the exact menu label may differ slightly; verify in your install).
         2. Locate the toggle that controls "import quantity from 1C XML" (often labeled `Import offer quantity` or `Sync stock from 1C` in ExtendShopaholic's settings YAML).
         3. Set it to OFF on every site running this plugin.
         4. Verify by re-running the next 1C XML import and confirming `offer.quantity` does NOT change.
       - This plugin does NOT auto-disable that setting (D-13 — explicit out-of-band, no cross-plugin migration).

    9. **## Console command — recompute active flags from stock**
       - Command: `php artisan goodsreceived:recompute_active_from_stock {--chunk=500}`
       - When to run:
         - After bulk-importing stock outside the plugin (e.g., manual SQL update of `offer.quantity`)
         - After flipping `auto_deactivate_on_zero` or `auto_activate_on_stock` ON for the first time (back-fills the active state of offers that already had qty=0 or qty>0 before the toggle change)
         - As a periodic reconcile cron (recommended: weekly low-traffic window)
       - What it does: walks every offer in chunks of 500 (override via `--chunk=N`), reads current settings via `SettingsAccessor`, runs the same provenance-aware `ActiveFlagService::reconcile` logic the apply path uses. Skips offers where `active_managed_by='operator'` (operator-set deactivations are sticky).
       - Exit codes: 0 on success, 1 on any Throwable.
