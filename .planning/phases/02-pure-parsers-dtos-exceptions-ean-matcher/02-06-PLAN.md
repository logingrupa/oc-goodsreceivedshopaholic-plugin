<objective>
Resolve a batch of EAN strings to offer_ids in **exactly two queries** total, regardless of batch size. The two-pass strategy (offer.code first, product.code with single-offer guard second) is the locked behavior; tests assert query count via `DB::enableQueryLog()`. This plan also covers the QA-02 leading-zero EAN preservation guard and the defense-in-depth InvalidEan rejection.

Purpose: Per-line `Offer::where('code', $sEan)->first()` would issue N queries for N lines — fine for 5 rows, catastrophic for 500. Two-query batching is the difference between "import takes 200ms" and "import times out". And EAN as STRING (not int) is the difference between matching `'0000000012345'` correctly vs querying `12345`.
Output: 1 service class + 1 unit test file (QA-02 sub-tests rolled in per D-31).
</objective>

    **Function 1: matchBatch** (~35 lines):
    ```
    /**
     * @param list<string> $arEans
     * @return array<string, array{matched_offer_id: int|null, match_strategy: 'offer_code'|'product_code_single_offer'|'none'}>
     * @throws InvalidEanException
     */
    public function matchBatch(array $arEans): array
    {
        $this->assertAllEansValid($arEans);

        if ($arEans === []) {
            return [];
        }

        $arUnique = array_values(array_unique($arEans));

        // Pass 1: offer.code matches
        $arOfferRows = Offer::whereIn('code', $arUnique)->get(['id', 'code']);
        $arOfferMap = [];
        foreach ($arOfferRows as $obOfferRow) {
            $arOfferMap[(string) $obOfferRow->code] = (int) $obOfferRow->id;
        }

        $arUnmatched = array_values(array_diff($arUnique, array_keys($arOfferMap)));

        // Pass 2: product.code matches with single-offer guard
        $arProductMap = [];
        if ($arUnmatched !== []) {
            $arProductRows = Product::whereIn('code', $arUnmatched)
                ->has('offer', '=', 1)
                ->with(['offer:id,product_id'])
                ->get(['id', 'code']);
            foreach ($arProductRows as $obProductRow) {
                $obSingleOffer = $obProductRow->offer->first();
                if ($obSingleOffer === null) {
                    continue;  // safety: has('offer', '=', 1) should guarantee one, but null-guard PHPStan
                }
                $arProductMap[(string) $obProductRow->code] = (int) $obSingleOffer->id;
            }
        }

        // Build result for every input EAN (deduplicated keys ok; map preserves leading-zero strings)
        $arResult = [];
        foreach ($arUnique as $sEan) {
            if (isset($arOfferMap[$sEan])) {
                $arResult[$sEan] = ['matched_offer_id' => $arOfferMap[$sEan], 'match_strategy' => 'offer_code'];
                continue;
            }
            if (isset($arProductMap[$sEan])) {
                $arResult[$sEan] = ['matched_offer_id' => $arProductMap[$sEan], 'match_strategy' => 'product_code_single_offer'];
                continue;
            }
            $arResult[$sEan] = ['matched_offer_id' => null, 'match_strategy' => 'none'];
        }

        return $arResult;
    }
    ```

    **Function 2: assertAllEansValid** (private, ~12 lines):
    ```
    /**
     * @param list<string> $arEans
     * @throws InvalidEanException
     */
    private function assertAllEansValid(array $arEans): void
    {
        foreach ($arEans as $sEan) {
            if (preg_match(self::EAN_REGEX, $sEan) !== 1) {
                throw new InvalidEanException(
                    \Lang::get('logingrupa.goodsreceivedshopaholic::lang.exception.invalid_ean'),
                    ['raw' => $sEan, 'expected_format' => '13 digits'],
                );
            }
        }
    }
    ```

    Why pre-validate at the matcher boundary even though parser already filtered (D-16)? Defense-in-depth (security threat T-02-06-01). The parser's lenient skip (Log::warning + skipped_rows) means an invalid EAN should NEVER reach `matchBatch`. If it ever does, an exception thrown here is louder and more diagnostic than a silent miss.

    **Function 3: buildMatchedLines** (~12 lines):
    ```
    /**
     * @param list<ParsedLine> $arLines
     * @param array<string, array{matched_offer_id: int|null, match_strategy: string}> $arMatchMap
     * @return list<MatchedLine>
     */
    public function buildMatchedLines(array $arLines, array $arMatchMap): array
    {
        $arResult = [];
        foreach ($arLines as $obLine) {
            $arEntry = $arMatchMap[$obLine->ean] ?? ['matched_offer_id' => null, 'match_strategy' => 'none'];
            /** @var 'offer_code'|'product_code_single_offer'|'none' $sStrategy */
            $sStrategy = $arEntry['match_strategy'];
            $arResult[] = new MatchedLine(line: $obLine, matched_offer_id: $arEntry['matched_offer_id'], match_strategy: $sStrategy);
        }
        return $arResult;
    }
    ```

    The `@var` annotation narrows the strategy string for level-10 (MatchedLine constructor expects the literal-string union per plan 02-01).

    Hungarian throughout: `$arEans`, `$arUnique`, `$arUnmatched`, `$arOfferRows`, `$arOfferMap`, `$arProductRows`, `$arProductMap`, `$arResult`, `$obOfferRow`, `$obProductRow`, `$obSingleOffer`, `$obLine`, `$sEan`, `$sStrategy`, `$arEntry`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Parsed EAN strings → SQL `whereIn` clause | EAN values originated from attacker-controlled HTM. They are now bound parameters in an Eloquent `whereIn` clause that talks to the production DB. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06-01 | Tampering (SQL injection via EAN) | `Offer::whereIn('code', $arEans)` | mitigate | Eloquent `whereIn` parameterizes the IN clause via PDO bind variables — values are never string-concatenated into SQL. Defense-in-depth: `assertAllEansValid` regex-rejects anything not exactly 13 digits BEFORE the query is built. Test asserts InvalidEan throw with zero queries logged. |
| T-02-06-02 | Information Disclosure (timing oracle) | EAN-not-found vs EAN-found timing differences | accept | Negligible value — attacker would learn whether a 13-digit EAN exists in the operator's product catalog, which is publicly observable from the storefront anyway. |
| T-02-06-03 | Denial-of-Service (huge whereIn list) | matchBatch with millions of EANs | accept | Bounded upstream by Phase 4 controller (HTM upload size cap → MAX_ROWS = 10000 in parser → at most 10000 EANs reach the matcher). MySQL/MariaDB IN clauses with 10K params are well within `max_allowed_packet`. |
| T-02-06-04 | Tampering (silent leading-zero stripping) | Implementation int-casts EAN somewhere | mitigate | QA-02 PreservesLeadingZeroEanTest pins the string-key invariant. If implementation drift int-casts an EAN, the test fails (`'0000000012345'` becomes `12345`). |
</threat_model>
