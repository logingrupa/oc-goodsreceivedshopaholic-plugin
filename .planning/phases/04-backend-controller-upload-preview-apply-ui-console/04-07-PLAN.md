A) Create `tests/unit/Controllers/ControllerTestCase.php`:

Pure abstract base class (PHP — not a Pest file). Extends `ApplyTestCase` so it inherits the hermetic Apply schema. Adds:

```php
<?php

declare(strict_types=1);

require_once __DIR__.'/../Apply/ApplyTestCase.php';

use Symfony\Component\HttpFoundation\File\UploadedFile;

/**
 * Shared base for tests/unit/Controllers/* (Phase 4 plans 04-04..04-07).
 *
 * Adds:
 *   - BackendAuth boundary fakes (sanctioned per CLAUDE.md / Tiger-Style: BackendAuth
 *     is a permission boundary, not business logic; mocking is allowed at boundaries).
 *   - UploadedFile factory helper for synthetic file uploads from fixtures.
 *   - Per-permission grant helpers: grantPermission(string $sKey) sets up a fake user
 *     with that one permission; revokeAllPermissions() sets up a fake user with none.
 *
 * setUp(): inherits ApplyTestCase's schema bootstrap. afterEach: \Mockery::close()
 * (Pest's default Mockery integration handles this when present, but explicit close
 * survives Pest version drift).
 */
abstract class ControllerTestCase extends ApplyTestCase
{
    protected function tearDown(): void
    {
        \Mockery::close();
        parent::tearDown();
    }

    /**
     * Configure BackendAuth so userHasAccess returns true ONLY for
     * $sPermissionKey, false for everything else.
     */
    protected function grantPermission(string $sPermissionKey): void
    {
        \BackendAuth::shouldReceive('userHasAccess')
            ->with($sPermissionKey)
            ->andReturn(true);
        \BackendAuth::shouldReceive('userHasAccess')
            ->withAnyArgs()
            ->andReturn(false);
        \BackendAuth::shouldReceive('getUser')
            ->andReturn((object) ['id' => 99]);
    }

    /**
     * Configure BackendAuth so userHasAccess returns false for ALL permissions.
     */
    protected function revokeAllPermissions(): void
    {
        \BackendAuth::shouldReceive('userHasAccess')
            ->withAnyArgs()
            ->andReturn(false);
        \BackendAuth::shouldReceive('getUser')
            ->andReturn((object) ['id' => 99]);
    }

    /**
     * Create a synthetic Symfony UploadedFile from a fixture path. Used to
     * drive Input::file() bindings without spinning a real HTTP request.
     */
    protected function makeUploadedFile(string $sFixturePath, string $sClientName, ?string $sExtension = null): UploadedFile
    {
        $sExt = $sExtension ?? pathinfo($sFixturePath, PATHINFO_EXTENSION);
        $sMime = strtolower($sExt) === 'htm' ? 'text/html' : 'application/octet-stream';

        return new UploadedFile($sFixturePath, $sClientName, $sMime, null, true /* test mode */);
    }
}
```

The `withAnyArgs()->andReturn(false)` LATER expectation is Mockery's "default" — earlier `with($sPermissionKey)->andReturn(true)` matches first, the catch-all returns false for everything else.

Hard rules: declare(strict_types=1), abstract class, Hungarian, decision link D-44 (backend auth simulation rationale).

NOTE: This file is a non-Pest helper, so PHPStan needs to be told to skip rule complaints about test-only artifacts. Test files are already in `excludePaths`: `tests/` per phpstan.neon — verify before write.

Out-of-scope for this plan — defer to a Phase 5 cleanup if it bothers anyone. The structural contract (permission gates exist) is independently pinned by THIS plan's 4 files; refactoring is purely DRY improvement.

A) `tests/unit/Controllers/RequiresUploadPermissionTest.php`:

```php
<?php

declare(strict_types=1);

use Logingrupa\GoodsReceivedShopaholic\Controllers\Invoices;
use October\Rain\Exception\AjaxException;

require_once __DIR__.'/ControllerTestCase.php';

uses(ControllerTestCase::class);

/**
 * QA-10 — Permission gate for onUpload (logingrupa.goodsreceived.upload_invoices).
 *
 * Three contract pins per the plan-04-07 behavior list. The handler is the
 * sole entry point that the multi-file upload widget hits; without this gate
 * any backend user could overwhelm the parser with arbitrary HTM payloads.
 */

it('rejects onUpload when operator has no permissions', function (): void {
    $this->revokeAllPermissions();

    expect(fn () => (new Invoices())->onUpload())->toThrow(AjaxException::class);
});

it('permits onUpload when operator holds upload_invoices permission', function (): void {
    $this->grantPermission('logingrupa.goodsreceived.upload_invoices');

    // No files provided yet → next-stage validation throws "no_files" — that
    // proves we got past the permission gate. Any AjaxException with
    // 'no_files' or no exception at all = pass; AjaxException with
    // 'forbidden' = fail the gate.
    try {
        (new Invoices())->onUpload();
    } catch (AjaxException $obException) {
        // Accept ONLY non-forbidden exceptions here.
        expect((string) $obException->getMessage())->not->toContain('permission');
    }
});

it('rejects onUpload when operator holds an unrelated permission only', function (): void {
    $this->grantPermission('logingrupa.goodsreceived.apply_invoices'); // wrong key

    expect(fn () => (new Invoices())->onUpload())->toThrow(AjaxException::class);
});
```

The 'permits' test is subtle: Pest's `expect(...)->toThrow(...)` would fail any exception even if it isn't the gate exception we want to assert against. The try/catch + `expect($e->getMessage())->not->toContain('permission')` lets us pass through validation exceptions that happen AFTER the gate (the 'No files were uploaded' AjaxException from onUpload's `if (! is_array($arFiles))` guard).

B) `tests/unit/Controllers/RequiresApplyPermissionTest.php` — same 3-test structure, targeting `onApply`. The handler hits `Input::get('invoice_id')` early after the permission check; with no input, will throw 'invoice_id is required' AjaxException. Same accept-condition: AjaxException whose message does NOT contain 'permission' = passed the gate.

C) `tests/unit/Controllers/RequiresOverridePermissionTest.php` — targets `onOverrideConfirm`. The handler hits `Input::get('confirm_typed')` after the gate; without input, throws 'Type OVERRIDE exactly to confirm.' AjaxException.

D) `tests/unit/Controllers/RequiresInitialResetPermissionTest.php` — targets `onInitialResetConfirm`. Same pattern.

For ALL 4 files:
- Header: `<?php declare(strict_types=1);`
- `require_once __DIR__.'/ControllerTestCase.php';`
- `uses(ControllerTestCase::class);`
- Class docblock contains `QA-10` + the specific permission key + the 3 pin contracts.

Hard rules:
- Hungarian throughout test bodies.
- No `try { } catch { }` swallowing — every `try` either rethrows or asserts on the caught exception.
- Decision link in each file docblock: QA-10 / D-36 / D-37 / D-44.
- Pint clean.
- The 'forbidden' substring is the controller's `flash.forbidden` lang string body. Assertion is on `expect(...)->toContain('permission')` from the lang body — verify the lang key body in 04-04. If the controller uses `assertPermission` AjaxException with 'You do not have permission to perform this action.', the substring check should be `->not->toContain('permission')` to pass through non-permission exceptions.

Test the 'permits' path more carefully:

```php
it('permits onApply when operator holds apply_invoices permission', function (): void {
    $this->grantPermission('logingrupa.goodsreceived.apply_invoices');

    $obException = null;
    try {
        \Input::merge(['invoice_id' => 0]); // intentional invalid id, fires later validation
        (new Invoices())->onApply();
    } catch (AjaxException $obCaught) {
        $obException = $obCaught;
    }

    // Permission-gate exceptions must NOT have been thrown — only
    // downstream validation exceptions are acceptable here.
    if ($obException !== null) {
        expect((string) $obException->getMessage())->not->toContain('permission');
    }
});
```

This passes if EITHER the handler reaches a downstream validation error (acceptable — gate was passed) OR throws no exception at all (also acceptable — gate was passed). Fails ONLY if the gate exception fires.
