use Cache;
use Logingrupa\GoodsReceivedShopaholic\Classes\Dto\ApplyResult;
use Logingrupa\GoodsReceivedShopaholic\Classes\Exception\ApplyAlreadyDoneException;
use Logingrupa\GoodsReceivedShopaholic\Classes\Orchestrator\ApplyOrchestrator;

```php
private const APPLY_LOCK_TTL_SECONDS = 60;

/**
 * AJAX handler: render the Apply confirmation modal listing total units,
 * matched offer count, and unmatched line count for an invoice the
 * operator is about to apply (UI-04 / D-12).
 *
 * @return array<string, string>
 *
 * @throws AjaxException
 */
public function onApplyShowConfirm(): array
{
    $this->assertPermission('logingrupa.goodsreceived.apply_invoices');

    $iInvoiceId = (int) Input::get('invoice_id');
    $obInvoice = Invoice::find($iInvoiceId);
    if (! $obInvoice instanceof Invoice) {
        throw new AjaxException(['message' => sprintf('Invoice #%d not found.', $iInvoiceId)]);
    }

    $iTotalUnits = (int) InvoiceLine::where('invoice_id', $iInvoiceId)
        ->whereNotNull('matched_offer_id')
        ->sum(\DB::raw('COALESCE(override_qty, qty)'));

    return [
        '#applyConfirm' => $this->makePartial('_partials/_apply_confirm', [
            'invoice'       => $obInvoice,
            'total_units'   => $iTotalUnits,
            'offer_count'   => (int) $obInvoice->matched_lines,
            'unmatched_count' => (int) $obInvoice->unmatched_lines,
        ]),
    ];
}

/**
 * AJAX handler: execute apply via ApplyOrchestrator. Cache::lock prevents
 * double-click double-apply (D-13). The orchestrator itself uses
 * Invoice::lockForUpdate() inside its DB::transaction (Phase 3 D-24) — the
 * Cache::lock here is a CHEAPER first line of defense that fails fast on
 * the SECOND click instead of waiting for the DB row lock to release.
 *
 * @return array<string, string>
 *
 * @throws AjaxException
 */
public function onApply(): array
{
    $this->assertPermission('logingrupa.goodsreceived.apply_invoices');

    $iInvoiceId = (int) Input::get('invoice_id');
    if ($iInvoiceId <= 0) {
        throw new AjaxException(['message' => 'invoice_id is required.']);
    }

    $iUserId = $this->resolveBackendUserId();

    $obLock = Cache::lock(sprintf('apply-invoice-%d', $iInvoiceId), self::APPLY_LOCK_TTL_SECONDS);
    if (! $obLock->get()) {
        return [
            '#applyResult' => $this->makePartial('_partials/_apply_in_progress'),
        ];
    }

    try {
        $obOrchestrator = app(ApplyOrchestrator::class);
        if (! $obOrchestrator instanceof ApplyOrchestrator) {
            throw new \RuntimeException('ApplyOrchestrator failed to resolve.');
        }
        try {
            $obResult = $obOrchestrator->apply($iInvoiceId, $iUserId);
            return [
                '#applyResult' => $this->makePartial('_partials/_apply_success', [
                    'invoice_id' => $iInvoiceId,
                    'result'     => $obResult,
                ]),
            ];
        } catch (ApplyAlreadyDoneException $obException) {
            return [
                '#applyResult' => $this->makePartial('_partials/_apply_already_done', [
                    'context' => $obException->arContext,
                ]),
            ];
        }
    } finally {
        $obLock->release();
    }
}
```

Tiger-Style notes:
- The `try/finally` is the load-bearing pattern (D-13 / T-04-05-02 mitigation): even if ApplyOrchestrator throws ANY exception (including ApplyAlreadyDoneException), the finally releases the cache lock. The inner try/catch around `$obOrchestrator->apply` is for the typed exception only — every other exception propagates with the lock still released by the outer finally.
- L10 instanceof guard for `app()` (D-03-07-03 pattern).
- Function bodies are ~30 lines each — under the 70-line cap.
- `Cache::lock`'s `->get()` returns bool; `->release()` is idempotent — safe to call even if `->get()` failed (no-op).
- COALESCE is used in onApplyShowConfirm because the actual stock that lands is `override_qty ?? qty` per StockApplyService (Phase 3 03-03). Using DB::raw('COALESCE(override_qty, qty)') avoids the N+1 hydration cost.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| onApply AJAX | mutates production stock — must be permission-gated, lock-debounced, and orchestrator-routed |
| Cache::lock | Laravel's atomic lock primitive backed by file/database/redis cache; lock key derived from invoice_id (operator-controllable but bounded) |
| ApplyOrchestrator | already lockForUpdate-protected at the DB layer (Phase 3 D-24); Cache::lock here is a CHEAPER first defense |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05-01 | Tampering | concurrent double-click double-applies stock | mitigate | Cache::lock fails fast on second click; if cache lock somehow misses, ApplyOrchestrator's lockForUpdate + status='applied' check catches at DB layer |
| T-04-05-02 | Denial of Service | exception inside try block leaks the lock | mitigate | finally { $lock->release(); } releases the lock regardless of throw path; pinned by 'failing orchestrator releases lock' test |
| T-04-05-03 | Elevation of privilege | unauthorized user calls onApply | mitigate | assertPermission at handler entry; per-action gate ('apply_invoices' permission) on top of class-level loose gate |
| T-04-05-04 | Information disclosure | ApplyAlreadyDoneException reveals prior_applied_by user_id | accept | the operator workflow needs this info to understand why their click did nothing; low-value disclosure (user_id is an int, not PII) |
| T-04-05-05 | Repudiation | apply commit not audit-logged | mitigate | ApplyOrchestrator::apply already calls ImportAuditService::logApply INSIDE the tx (Phase 3 D-24 step 7); controller does NOT need a separate log call |
| T-04-05-06 | Tampering | invoice_id manipulation to apply someone else's parsed invoice | accept | acceptable: any operator with apply_invoices permission can apply any parsed invoice. Per-invoice ownership is out of scope (not in REQUIREMENTS.md); the audit trail records WHO applied via applied_by_user_id |
</threat_model>
