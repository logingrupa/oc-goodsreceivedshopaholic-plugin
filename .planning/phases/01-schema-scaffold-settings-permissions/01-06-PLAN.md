<reference_register_permissions>
**October's `registerPermissions()` contract** (from `Backend\Classes\AuthManager::registerPermissions`):

Returns an array keyed by permission code, each value being an array with:
- `label`: human-readable label (string or lang key)
- `tab`: optional tab grouping label (string or lang key) — permissions sharing the same tab are visually grouped
- `comment`: optional description rendered as helper text in the picker
- `roles`: optional array of role codes that auto-grant this permission (typically omitted; let admins assign explicitly)
- `order`: optional integer for sort order within the tab

Permission codes are namespaced strings using dots. Convention: `<vendor>.<plugin-shortname>.<action>` — so our codes are `logingrupa.goodsreceived.<action>`.

Example pattern from October core / common usage:
```php
public function registerPermissions(): array
{
    return [
        'logingrupa.goodsreceived.upload_invoices' => [
            'label' => 'logingrupa.goodsreceivedshopaholic::lang.permission.upload_invoices',
            'tab'   => 'logingrupa.goodsreceivedshopaholic::lang.permission.tab',
            'comment' => 'logingrupa.goodsreceivedshopaholic::lang.permission.upload_invoices_comment',
            'order' => 100,
        ],
        // ... 3 more
    ];
}
```

The `tab` value MUST be a translation key (or static label) — October groups permissions by exact-match `tab` string, so all 4 of our entries MUST use the IDENTICAL `tab` value.
</reference_register_permissions>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend role -> permission gate -> action | Each backend user has 0..N permissions assigned via roles. Phase 4 controllers gate actions by these permissions. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-23 | Elevation of Privilege | `run_initial_reset` permission | mitigate | Splitting `run_initial_reset` into its own permission (NOT bundled with `apply_invoices`) lets admins grant routine apply rights to operators while reserving the destructive baseline-reset action for owner-level roles. Fine-grained least-privilege. |
| T-01-24 | Spoofing | permission code naming collision | mitigate | All 4 codes use the prefix `logingrupa.goodsreceived.` — globally unique within an October install (no other plugin uses this vendor.shortname combination, verified by grep across plugins/). |
| T-01-25 | Repudiation | who-did-what audit | accept | Phase 4 controller will record `applied_by_user_id` on Invoice rows when Apply runs. The permission gate adds RBAC enforcement; audit trail comes from the controller layer. |
| T-01-26 | Tampering | role -> permission assignment | accept | RBAC assignment is itself a backend action gated by `system::manage_users` (October core permission). Outside this plugin's threat model — assumed properly configured by deployment. |
| T-01-27 | Information Disclosure | tab label / comment leakage | accept | Permission tab labels and comments are by design visible to anyone who can view the role-edit form. No secrets in any of the lang values for `permission.*` (Plan 03 verified — all are descriptive English text). |
| T-01-28 | Denial of Service | over-grant via 'roles' auto-assignment | mitigate | NO `roles` key on any of our 4 permissions (acceptance criterion #8 grep-gates this). Admins must explicitly assign — prevents new roles from accidentally inheriting destructive permissions. |
</threat_model>
