```php
<?php

declare(strict_types=1);

namespace Logingrupa\GoodsReceivedShopaholic\Models;

use October\Rain\Database\Traits\Multisite;
use System\Models\SettingModel;

/**
 * Class Settings
 *
 * @package Logingrupa\GoodsReceivedShopaholic\Models
 *
 * Backend settings model for the Goods Received Notes plugin.
 * Per-site isolation provided by October Rain's `Multisite` trait â€” same mechanism
 * used internally by `Lovata\Toolbox\Models\CommonSettings`. Extends `SettingModel`
 * directly per locked decision D15 (NOT `CommonSettings`), but adds the multisite
 * trait inline so per-site isolation behavior is functionally identical.
 *
 * @property bool $enabled
 * @property bool $auto_deactivate_on_zero
 * @property bool $auto_activate_on_stock
 * @property bool $allow_initial_reset
 *
 * @method static \October\Rain\Database\Builder|Settings newQuery()
 *
 * @mixin \System\Models\SettingModel
 * @mixin \Eloquent
 */
class Settings extends SettingModel
{
    use Multisite;

    public const SETTINGS_CODE = 'logingrupa_goodsreceivedshopaholic_settings';

    /** @var string */
    public $settingsCode = self::SETTINGS_CODE;

    /** @var string */
    public $settingsFields = 'fields.yaml';

    /**
     * Multisite trait requires `$propagatable` to be an array (initializeMultisite()
     * throws otherwise). Empty list = no attribute is auto-synced across sites;
     * each site's settings row is fully independent. This is what we want for
     * per-site toggles.
     *
     * @var array<int, string>
     */
    protected $propagatable = [];
}
```

Implementation notes:
- `use Multisite;` â€” the trait that gives Settings models per-site isolation. Verified at `vendor/october/rain/src/Database/Traits/Multisite.php`. Not to be confused with `Lovata\Toolbox\Traits\Models\MultisiteHelperTrait` which is a different trait designed for content models with `site()` morphToMany relations (NOT for Settings).
- `protected $propagatable = []` â€” REQUIRED by the trait. The trait's `initializeMultisite()` method throws `Exception('The $propagatable property in <class> must be an array...')` if missing or non-array. Empty array means: no attribute syncs across sites; each site has fully independent settings rows. This matches our intent (per-site toggles).
- `SETTINGS_CODE` exposed as `public const` for type-safe references in tests.
- `@property bool` lines on each toggle let PHPStan level 10 type-check `$obSettings->enabled` correctly.
- `@mixin \System\Models\SettingModel` brings in the static `::get($sCode, $mDefault = null)` and `::set($sCode, $mValue)` methods for PHPStan.
- NO RainLab.Translate behavior implemented (CommonSettings adds `$implement = ['@RainLab.Translate.Behaviors.TranslatableModel']`). Reason: our 4 switches are booleans, not translatable strings â€” no fields need cross-locale storage. Toggles are stored as `bool` per site. Adding the behavior would be cargo-culted.
- NO `extend()` or event subscribers in this file. Settings model is pure data.

D15 reconciliation: see `<objective>` section above. The literal CONTEXT.md class names (`MultisiteInterface`, `MultisiteHelperTrait` from wrong path) do NOT exist in this codebase. We use the trait that actually delivers per-site isolation for Settings models, matching what `CommonSettings` itself uses.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend authenticated user -> Settings table | Backend operator can flip toggles. Untrusted toggle values cannot exist (October casts to bool); risk is misconfiguration, not injection. |
| Settings::get() -> service decision logic | Phase 3 services read these toggles. Wrong default could expose unsafe behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-18 | Tampering | `Settings` field values | mitigate | All 4 fields are `type: switch` with `default: false`. October casts switch values to bool on save. Even if an operator submits arbitrary form data, the cast prevents enum-like values from being stored. |
| T-01-19 | Elevation of Privilege | `allow_initial_reset` toggle | mitigate | Default `false` means the destructive initial-reset flow is OFF until an operator with backend access explicitly enables it. Phase 3 `InitialResetService` will additionally require the `run_initial_reset` permission (created in plan 01-06) AND a typed `RESET` confirmation in the UI (Phase 4). Defense in depth. |
| T-01-20 | Information Disclosure | per-site toggle leakage | mitigate | `October\Rain\Database\Traits\Multisite` enforces a global query scope that filters by `site_id`. A bug here would leak settings across sites. Plan 01-07 adds `IsMultisiteAwareTest` and `MultisiteContextSwitchClearsCacheTest` to prove this works in practice. |
| T-01-21 | Denial of Service | unbounded `propagatable` array | mitigate | `protected $propagatable = []` â€” empty by design. The Multisite trait's `multisiteAfterCreate` would otherwise enumerate this list to sync attributes; empty list = O(1). |
| T-01-22 | Repudiation | toggle change audit | accept | October's `SettingModel` has built-in `created_at`/`updated_at` (when set on the underlying `system_settings` table). Per-site toggle change events can be reconstructed from these timestamps. No additional audit needed at Phase 1. |
</threat_model>
