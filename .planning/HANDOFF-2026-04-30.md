## Outstanding Production Bugs (priority order)

### BUG 1 — `_button_` partial not found at `/invoices/update/{id}`

**Error:** `The partial '_button_' is not found.` at `modules/system/traits/ViewMaker.php:97` when opening invoice detail page.

**Likely cause:** A `columns.yaml` field uses `partial: 'button'` or similar with leading/trailing underscore confusion, OR `update.htm` references an empty/missing partial reference. October prepends `_` to base name — passing `'button'` looks for `_button.htm`. If something passes `'button_'` October looks for `_button_.htm`.

**Investigate:**
```bash
grep -rn "button_\|partial:.*button\|makePartial.*button" controllers/ models/ Plugin.php
```

Likely culprits:
- `controllers/invoices/update.htm` line with form-buttons section
- `models/invoice/columns.yaml` if any column has `partial: button`
- `models/invoice/fields.yaml` `formAction` or button fields

**Fix:** correct the partial name OR create the missing `_button.htm` file.

### BUG 2 — `Apply to stock` modal renders inline, not in popup

`data-control="popup"` + `data-handler="onApplyShowConfirm"` is supposed to render `_apply_confirm.htm` in October's popup widget. Instead it lands inline on the page.

**Cause:** controller `onApplyShowConfirm` returns `['#applyConfirm' => $sHtml, 'partial' => $sHtml, 'result' => $sHtml]`. October's popup widget reads only ONE response key (likely `result` or unkeyed body). The selector `#applyConfirm` may be triggering October's selector-update which injects content into the empty inline anchor BEFORE popup widget consumes it. Also `_upload_form.htm` no longer has anchor divs (we dropped them) but `upload.htm` may STILL have one in `<div class="layout-row" id="applyResult">`.

**Investigate the popup widget contract:**
```bash
grep -rn "data-control=\"popup\"" /home/forge/nailscosmetics.lv/modules/backend/widgets/ /home/forge/nailscosmetics.lv/plugins/lovata/ | head -10
```

Find a working reference plugin example (RainLab.User, Lovata.Shopaholic).

**Fix path:** Either:
- Return ONLY the partial string (not array) from `on*ShowConfirm` handlers — popup widget consumes response body directly
- OR: keep array shape but change to `['result' => $sHtml]` (single key, well-known popup contract)
- Verify by grepping October source for `data-control="popup"` rendering logic

### BUG 3 — UI is "ugly" — use October-native widgets

User feedback: "why not using octobercms components for UI and form generation?"

**Current state (anti-pattern):**
- `<input type="file" multiple>` raw HTML in `_upload_form.htm`
- Custom inline-styled `.gr-tabs` CSS for tab nav
- `<table class="data-table">` raw markup in `_preview_lines.htm` (already simplified to compact card; see commit `d2c0ef7`)
- Custom `<div class="modal-dialog">` in `_apply_confirm.htm`, `_override_confirm.htm`, `_initial_reset_confirm.htm`

**Target (October-native — RainLab.User pattern):**
- Use `Backend\FormWidgets\FileUpload` for `original_file` attachOne (already in Invoice model)
- Use `Backend\FormController` widgets via `fields.yaml` for the upload form
- Use `<div data-control="loading-indicator">` instead of custom upload-progress span
- Use October's `Flash::success(...)` / `Flash::error(...)` helpers
- For modals: use `popup` widget with returned partial (no custom `<div class="modal-dialog">` wrappers — popup wraps automatically)
- For tabs: use `<div data-control="tab">` + `<ul class="nav nav-tabs">` with backend's bundled CSS class `oc-tabs` if available, OR keep inline-CSS `.gr-tabs` (acceptable for small custom UI)

**Reference:** `plugins/rainlab/user/controllers/Users.php` + `controllers/users/_*.htm` for FormController + RelationController + popup conventions.
